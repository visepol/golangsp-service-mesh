Não é o proxy, é a camada.

A pilha OSI define os limites do que a infraestrutura consegue ver.

L1 — Física
L2 — Enlace
L3 — Rede
L4 — Transporte
TCP/UDP — enxerga conexões
L5 — Sessão
L6 — Apresentação
L7 — Aplicação
HTTP, gRPC — enxerga requests individuais

O que o Ingress enxerga em L4

Ele entrega pacotes TCP baseados em IP e Porta. O conteúdo (HTTP/2) é um bloco opaco de bytes cifrados.

Não é o nginx.
Não é a configuração.

L4 vs L7

L4 enxerga a conexão.

L7 enxerga o request.

Service Mesh

Control Plane (Istiod)

  • Interage com a API do Kubernetes
  • Injeta o sidecar proxy em cada pod
  • Distribui as regras de roteamento

Data Plane (Envoy)

  • Roda junto com a aplicação no pod
  • Intercepta todo o tráfego de entrada e saída
  • Opera em L7 — enxerga o request

Por que isso não é over-engineering

O que você paga

  • Sidecar por pod (CPU/Mem extra)
  • Control plane no cluster
  • Curva de aprendizado

O que você ganha

  • Load balancing por request
  • mTLS automático
  • Circuit breaking & Retry
  • Sem uma linha de código Go.

Algoritmos de Balanceamento em L7

Aplicados diretamente nas conexões HTTP/2 ativas.

ROUND_ROBIN

Distribui sequencialmente por request. Aceita pesos.

LEAST_CONN

Envia para o pod com menos requests ativos no momento.

RANDOM

Distribuição aleatória padronizada.

Controle via YAML (DestinationRule + VirtualService)

# DestinationRule kind: DestinationRule spec: host: go-api trafficPolicy: loadBalancer: simple: ROUND_ROBIN subsets: - { name: v1, labels: { version: v1 } } - { name: v2, labels: { version: v2 } } - { name: v3, labels: { version: v3 } } --- # VirtualService kind: VirtualService spec: hosts: [go-api] http: - route: - destination: { host: go-api, subset: v1 } weight: 60 - destination: { host: go-api, subset: v2 } weight: 20 - destination: { host: go-api, subset: v3 } weight: 20
