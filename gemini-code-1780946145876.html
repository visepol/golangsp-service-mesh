A evolução necessária

Um request por conexão.
Overhead alto de handshakes TCP/TLS para cada chamada.

HTTP/2 (Padrão Go)

Multiplexação. Vários requests fluindo paralelamente sobre uma única conexão TCP.

A vantagem que vira armadilha

Com HTTP/2, o handshake acontece apenas uma vez.

Milhares de requests subsequentes do mesmo cliente usam a mesma conexão já estabelecida.

O que a infraestrutura enxerga

A Perspectiva do Ingress (L4)

Ele distribui conexões, não requests. Como há apenas 1 conexão HTTP/2 aberta, ela é roteada inteira para 1 pod.

Conexão TCP #1982 → Pod 1

A realidade nos Pods

A média de CPU de 33% esconde um problema crítico.

Pod 2
Pod 3

Escalar horizontalmente não resolve.
A conexão já está estabelecida.

• Como aplicar retries de forma inteligente?

Não é o proxy, é a camada.

A pilha OSI define os limites do que a infraestrutura consegue ver.

L1 - L3
Física, Enlace, Rede
L4 (Transporte)
TCP/UDP. Onde o Nginx padrão e os Load Balancers da nuvem operam.
L5 - L6
Sessão, Apresentação
L7 (Aplicação)
HTTP/gRPC. Enxerga headers, paths e requests individuais.

O que o Ingress enxerga em L4

Ele entrega pacotes TCP baseados em IP e Porta. O conteúdo (HTTP/2) é um bloco opaco de bytes cifrados.

[ENCRYPTED TCP PAYLOAD 10101010...]

Não é um erro de configuração.
É o limite da camada.

L4 vs L7

L4 enxerga a conexão.

L7 enxerga o request.

Service Mesh (Istio)

Control Plane (Istiod)

  • Interage com a API do Kubernetes
  • Injeta o proxy automaticamente
  • Distribui as regras de roteamento

Data Plane (Envoy)

  • Roda junto com a aplicação no pod
  • Intercepta todo o tráfego I/O
  • Opera em L7 — enxerga o request

Não é over-engineering. É arquitetura.

O que você paga

  • Sidecar por pod (CPU/Mem extra)
  • Control plane no cluster
  • Curva de aprendizado

O que você ganha

  • Load balancing por request
  • mTLS automático
  • Circuit breaking & Retry
  • Sem tocar em 1 linha de código Go.

Algoritmos de Balanceamento em L7

Aplicados diretamente nas conexões HTTP/2 ativas.

DEMO

ROUND_ROBIN

Distribui sequencialmente por request. Aceita pesos.

LEAST_CONN

Envia para o pod com menos requests ativos no momento.

RANDOM

Distribuição aleatória padronizada.

Controle via YAML (VirtualService)

apiVersion: networking.istio.io/v1alpha3 kind: VirtualService metadata: name: go-api-routing spec: hosts: - "go-api.prod.svc.cluster.local" http: - route: - destination: host: go-api subset: v1 weight: 60 - destination: host: go-api subset: v2 weight: 20 - destination: host: go-api subset: v3 weight: 20

O Impacto Arquitetural

Padrão K8s (L4)

1 Conexão = 1 Pod a 100%

Service Mesh (L7)

1 Conexão = Requests Distribuídos (60/20/20)

Seu código Go cuida da lógica de negócio.
A infraestrutura cuida do tráfego.

A solução para roteamento, retry e resiliência não é mais código.
